# Подробный процесс разработки

## Целевая аудитория
* Трейдеры и инвесторы, которым важно оперативно узнавать о движении монет

## Задачи системы
* Автоматический сбор цен
* Анализ аномалий на рынке
* Уведомление пользователей

## Описание процесса
* **Вход:** API биржи
* **Механизм:** Python + ML
* **Выход:** Telegram

### 📦 Группа 1: Рыночные данные и Аналитика

#### 1. Таблица `assets` (Активы)
Это неизменяемый справочник криптовалют, с которыми умеет работать система
* **`id` (integer, Primary Key):** Уникальный числовой идентификатор монеты. Используется для связи с другими таблицами, чтобы не дублировать текстовые названия в БД
* **`ticker` (varchar):** Биржевой тикер монеты (например, `XRP`, `BTC`). Используется модулем сбора данных для запросов к API биржи
* **`full_name` (varchar):** Полное название монеты (например, `Bitcoin`). Используется для вывода понятного текста пользователю в Telegram

#### 2. Таблица `price_history` (История цен)
Сюда ежеминутно записываются сырые рыночные данные. На основе этих данных работает алгоритм поиска аномалий
* **`id` (integer, Primary Key):** Уникальный номер записи о цене
* **`asset_id` (integer, Foreign Key):** Ссылка на `id` из таблицы `assets`. Указывает, к какому активу относится строка
* **`price` (float):** Цена монеты в данный момент времени
* **`volume` (float):** Объем торгов за последнюю минуту. Критически важен для анализа: резкий рост объема часто предвещает сильное движение цены
* **`timestamp` (timestamp):** Точная дата и время фиксации цены

#### 3. Таблица `analytics` (Анализ и Прогнозы)
Сюда скрипт-анализатор записывает результаты работы математических алгоритмов и ML-модели
* **`id` (integer, Primary Key):** Уникальный номер аналитического отчета
* **`asset_id` (integer, Foreign Key):** Ссылка на `id` из `assets` (для какого актива сделан расчет)
* **`predicted_direction` (varchar):** Краткосрочный прогноз направления тренда (`Up` — рост, `Down` — падение, `Stable` — боковик)
* **`is_anomaly` (boolean):** Главный триггер системы (`True/False`). Принимает значение `True`, если зафиксирован аномальный скачок цены или объема
* **`anomaly_description` (text):** Описание аномалии понятным языком (например: *"Резкий рост объема на 250% за 5 минут"*). Этот текст бот отправляет пользователю
* **`created_at` (timestamp):** Время проведения анализа

---

### 👤 Группа 2: Пользователи и подписки

#### 4. Таблица `users` (Пользователи)
Список всех пользователей, которые запустили Telegram-бота
* **`id` (integer, Primary Key):** Уникальный ID пользователя. Сюда сохраняется реальный **Telegram ID** человека, так как он уникален и не меняется
* **`username` (varchar):** Никнейм пользователя в Telegram (например, `@maxim`) для идентификации в системе
* **`registered_at` (timestamp):** Дата и время первого запуска бота

#### 5. Таблица `user_subscriptions` (Подписки пользователей)
Таблица связывает пользователей и активы, заменяя классический «будильник цен» на интеллектуальный мониторинг по уровням чувствительности
* **`id` (integer, Primary Key):** Уникальный номер подписки
* **`user_id` (integer, Foreign Key):** Ссылка на `id` из таблицы `users` (кто подписался)
* **`asset_id` (integer, Foreign Key):** Ссылка на `id` из таблицы `assets` (на что подписался)
* **`sensitivity_level` (varchar):** Уровень чувствительности (`Low` — уведомлять только при шоках, `Medium` — при сильных движениях, `High` — при любых колебаниях)
* **`is_active` (boolean):** Флаг активности подписки. Позволяет пользователю временно «поставить на паузу» уведомления без удаления самой подписки

---

### 💾 Группа 3: Логирование и Архивы

#### 6. Таблица `notifications_archive` (Архив уведомлений)
Хранит историю всех отправленных ботом сообщений. Нужна для аналитики активности и предотвращения спама (чтобы не отправлять одно и то же уведомление дважды)
* **`id` (integer, Primary Key):** Уникальный номер отправленного сообщения
* **`user_id` (integer, Foreign Key):** Ссылка на `id` из `users` (кому ушло сообщение)
* **`message_text` (text):** Текст отправленного пользователю уведомления
* **`sent_at` (timestamp):** Точное время отправки.

#### 7. Таблица `system_logs` (Системный журнал)
Изолированная таблица для логов технического состояния приложения. Помогает отслеживать ошибки API биржи или сбои Telegram-серверов. Ни с чем не связана, так как логирует работу всей системы глобально
* **`id` (integer, Primary Key):** Номер лога
* **`event_type` (varchar):** Тип события (`INFO` — штатная работа, `WARNING` — задержка ответа, `ERROR` — критическая ошибка)
* **`message` (text):** Технический текст ошибки или системного сообщения
* **`created_at` (timestamp):** Время записи лога
