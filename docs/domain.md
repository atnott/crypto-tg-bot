# Предметная область

Предметная область проекта — мониторинг рыночных аномалий криптовалют и
доставка уведомлений пользователям Telegram.

## Основные сущности

| Сущность | Описание |
| --- | --- |
| Пользователь | человек, который взаимодействует с системой через Telegram |
| Актив | отслеживаемая торговая пара или криптовалюта |
| Цена | запись о текущей цене и объёме торгов актива |
| Подписка | связь пользователя с активом, по которому он хочет получать сигналы |
| Аналитика | результат расчёта тренда и признака аномалии |
| Уведомление | сообщение, отправленное пользователю при обнаружении события |

Эти сущности отражены в базе данных через таблицы `users`, `assets`,
`price_history`, `subscriptions`, `analytics` и `notifications_archive`.

## Роли

| Роль | Действия |
| --- | --- |
| Пользователь Telegram | запускает бота, смотрит статус рынка, управляет подписками |
| Сборщик данных | получает котировки, сохраняет историю цен, запускает анализ |
| Аналитический модуль | классифицирует движение цены и признак аномалии |
| Telegram-бот | показывает интерфейс и доставляет сообщения пользователю |

## Сценарии использования

Основные сценарии системы:

* регистрация пользователя в Telegram-боте;
* просмотр текущего статуса рынка;
* подписка пользователя на актив;
* отписка пользователя от актива;
* сбор рыночных данных;
* анализ тренда;
* отправка уведомления при обнаружении аномалии.

Подробные связи акторов и сценариев должны быть раскрыты на Use-case
диаграмме. Порядок вызовов между компонентами должен быть показан на Sequence
диаграмме.

Возможные направления:

* `PUMP` — краткосрочный рост;
* `DUMP` — краткосрочное падение;
* `STABLE` — выраженного направления нет;
* `NEUTRAL` — данных недостаточно для анализа.

### Отправка уведомлений

Если анализ выявил аномальное изменение, система находит пользователей,
подписанных на соответствующий актив, отправляет им сообщение в Telegram и
сохраняет факт отправки в `notifications_archive`.

## Бизнес-правила

* пользователь создаётся один раз, повторная команда `/start` не дублирует
  запись;
* пользователь может подписаться только на актив из справочника `assets`;
* повторная подписка на тот же актив не создаёт дубликат;
* уведомления отправляются только пользователям, подписанным на актив;
* результат анализа сохраняется отдельно от исходной истории цен;
* архив уведомлений используется для аудита отправленных сообщений.

## Жизненный цикл данных

Сначала система получает рыночные данные, затем сохраняет их в историю,
рассчитывает аналитический результат, выбирает подписчиков и доставляет
уведомления пользователям. Детальный порядок обмена сообщениями фиксируется в
Sequence диаграмме.
